1. Overview

Bring Data operates a machine-learning lead scoring service for Brazilian businesses running paid traffic (B2B). Our clients capture leads through their landing pages; our ML models score each lead in real time; and we send qualified-lead conversion events to advertising platforms so the platforms can optimize bidding toward high-quality leads. Meta Conversions API (CAPI) is in production today; Google Ads is being added now.

This document describes the specific use of the Google Ads API by Bring Data, the operations performed, the runtime architecture (which is mostly served by a different Google API), authentication, expected volume, access control, and security posture.

2. Scope of Google Ads API use

The Google Ads API is used solely to perform one-time configuration of conversion actions on client accounts. Specifically:

FieldValue
API serviceConversionActionService (mutation, via GoogleAdsService)
OperationMutateConversionActions (create)
Resource type createdconversion_action (type=UPLOAD_CLICKS, category=QUALIFIED_LEAD)
Read operationsSingle GoogleAdsService.search query checking whether a conversion action with the same name already exists (idempotency check)
When invokedOnce per client account during onboarding
Where invokedFrom a developer workstation, by running the script create_google_ads_conversion_actions.py

Two conversion actions are created per client account:

  • LeadQualified — fires whenever the ML model classifies a captured lead as qualified, with a per-conversion value proportional to the score decile.
  • LeadQualifiedHighQuality — fires only for top-decile leads (D9–D10).

Why API-based creation is necessary

Our clients' Google Ads accounts have been migrated to the unified conversion-action setup. On migrated accounts, the Google Ads UI does not expose the path to create a bare UPLOAD_CLICKS conversion action — the "+ Create conversion action" wizard forces a Google Sheets data source connector and loops back to the connector even when "Skip and set up later" is selected. Manual UI creation is therefore impossible for our client accounts. The ConversionActionService (mutation) is the only available path. This API-based creation route uses MUTATION endpoints of the Google Ads API, not the deprecated UploadClickConversions endpoint.

What this implementation does NOT do

  • Query reports (no GoogleAdsService.search beyond the idempotency check above)
  • Modify bidding strategies
  • Modify campaigns, ad groups, ads, or keywords
  • Add or remove negative keywords
  • Manage budgets
  • Manage account access or linking
  • Upload conversion events (this runs through Data Manager API — see section 3)

3. Runtime architecture (NOT using the Google Ads API)

Actual event upload at runtime — i.e., sending lead conversion events to Google Ads when leads are scored — uses the Google Data Manager API (datamanager.googleapis.com/v1/events:ingest), not the Google Ads API.

This design follows Google's published guidance from the Google Ads Developer Blog (May 2026): "We don't recommend implementing new offline conversion workflows using the Google Ads API. Upgrade to the Data Manager API." The Data Manager API authenticates via service account (no developer token required) and uses the events:ingest endpoint to send events to the conversion actions previously created by the one-time script.

End-to-end data flow

[Client landing page] │ lead submit (POST /lead_capture, fields: contact + utm + gclid) ▼ [Bring Data backend on Cloud Run] │ ├── ML scoring (LightGBM models, latency ~80ms) │ ├── Persist to leads_capi table (Cloud SQL) │ ├── Decision: should_send_to_destination(lead, capi_config, dest) │ ├──▶ Meta Conversions API (existing, production) │ └──▶ Google Data Manager API: events:ingest (productDestinationId = conversion_action_id created previously via Google Ads API)

The Google Ads API is only invoked from the developer workstation by the script create_google_ads_conversion_actions.py during initial client onboarding. The Cloud Run runtime never imports the google-ads SDK and never makes Google Ads API calls.

5. Tool architecture and code

Bring Data developed all components in-house. We do not use any third-party tool with our Google Ads API token. Stack:

  • Python 3.11, official google-ads Python library (v27+)
  • Single-file CLI script: create_google_ads_conversion_actions.py (idempotent — re-runs do not duplicate)
  • Supports --dry-run via validate_only=True for safe rehearsal
  • Helper to mint the OAuth refresh token: google_ads_oauth_refresh_token.py

7. Access control

Only Bring Data's internal technical team has access to the Google Ads API credentials and to the script that calls the API. Credentials are not shared with clients or with external users. Client Google Ads accounts are linked to the Bring Data MCC for operational purposes (Bring Data team members access the MCC to monitor that conversion actions are firing correctly), but clients do not have access to Bring Data's API credentials.

8. Security and data handling

  • API credentials kept in environment variables, never in source control (V2/.env is git-ignored).
  • Logs redact PII (email/phone are hashed before any log emission for downstream destinations).
  • The system complies with the Brazilian General Data Protection Law (LGPD, Law 13.709/2018).
  • Privacy policy published at https://bring-data.com/privacy.html.
  • Internal documentation classifies the developer token as a secret; rotation procedure exists in the runbook.

9. Token lifecycle and policy compliance

  • The API contact email (ramon@bring-data.com) is monitored daily and will be kept up to date with the Google Ads API team.
  • We commit to responding promptly to all communications from the Google Ads API team.
  • We will not share the developer token, share API credentials, or use the token in ways inconsistent with the Google Ads API Terms and Conditions.
  • We will continue to follow Google's guidance to use the Data Manager API for runtime conversion event upload, and limit our Google Ads API usage to the scope described in section 2.

10. Summary

This is a minimal, well-scoped use of the Google Ads API — limited to one-time conversion action creation on client accounts. Runtime event upload uses the Data Manager API as Google recommends. We are applying for Basic Access because Test Access does not permit conversion action creation on production accounts, and the migrated-account UI does not expose an alternative path. The developer token, once Basic Access is granted and the conversion actions are created, has no ongoing role in our runtime traffic.
