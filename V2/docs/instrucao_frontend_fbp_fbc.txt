INSTRUÇÃO PARA DEV FRONT-END — FBP/FBC NO RAILWAY
====================================================

CONTEXTO (situação atual)
--------------------------
O sistema de lead scoring usa duas fontes de dados no Railway PostgreSQL:

1. Tabela "Lead" (Prisma/n8n)
   - Recebe os dados do formulário via n8n
   - Tem 12.363 registros
   - Tem colunas fbp e fbc, mas estão vazias na maioria dos casos
     porque o n8n não repassa esses campos hoje

2. Tabela "leads_capi" (SQLAlchemy/sendBeacon)
   - Recebe os dados do formulário via sendBeacon direto para a API
   - FBP/FBC 100% preenchidos
   - Usada como fonte intermediária para enriquecer o envio ao Meta CAPI

O problema: o fluxo principal de CAPI depende de cruzar as duas tabelas
por email para recuperar FBP/FBC. Isso é frágil — se o sendBeacon falhar
(fire-and-forget, sem retry), o lead fica sem FBP/FBC no envio para a Meta.

A nota de evento no Meta está em 9.2 justamente por causa do FBP/FBC.
Queremos garantir 100% de cobertura de forma direta e sem dependência dupla.


O QUE PRECISA MUDAR
--------------------
A tabela "Lead" no Railway já tem as colunas fbp e fbc criadas.
Basta o n8n passar esses valores ao criar o registro.

No formulário da Página 1, o fbp e fbc já são capturados pelo JavaScript:

    const fbp = getCookie('_fbp');
    const fbc  = getCookie('_fbc');

O que falta: incluir esses campos no payload enviado ao n8n.

Payload atual enviado ao n8n:
    {
      nome, email, telefone, tem_computador,
      utms: { source, medium, campaign, term, content },
      data_envio
    }

Payload após a mudança:
    {
      nome, email, telefone, tem_computador,
      utms: { source, medium, campaign, term, content },
      fbp: fbp || null,   <-- ADICIONAR
      fbc: fbc  || null,  <-- ADICIONAR
      data_envio
    }

No n8n: mapear os campos fbp e fbc recebidos para as colunas
homônimas ao criar o registro na tabela "Lead".


O QUE NÃO PRECISA MUDAR
-------------------------
- O sendBeacon para /webhook/lead_capture pode continuar existindo
  como fallback — não remover, apenas deixar de depender dele
- A URL da API (Cloud Run) não muda
- O fluxo de scoring e envio CAPI não muda


RESULTADO ESPERADO
-------------------
Após a mudança, a tabela "Lead" terá FBP/FBC em ~100% dos registros.
O pipeline de CAPI (/railway/process-pending) já lê fbp/fbc da tabela
"Lead" e envia para a Meta — sem precisar de nenhuma tabela intermediária.
